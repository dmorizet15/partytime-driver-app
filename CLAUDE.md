### P0 follow-up — offline auth (cold-start) — 2026-06-14, NOT committed

Without this, the route cache was unreachable on a force-close→relaunch-offline: auth init bounced the driver to `/login`, where sign-in (a network call) also failed. #### Hang fix — iOS standalone airplane-mode "Loading…" forever (2026-06-14, NOT committed)

The first cut deadlocked on iOS: force-close in airplane mode → relaunch → black "Loading…" that never resolved. Two compounding causes: (1) **`navigator.onLine` is unreliable on iOS standalone** — it can read `true` in airplane mode for a beat after launch, so the offline net (gated on `onLine === false`) never ran; (2) supabase's init **token-refresh hangs with no network timeout**, so `onAuthStateChange` never fired `INITIAL_SESSION` → the handler's `finally { setLoading(false) }` never ran. `loading` stayed `true` forever. There was **no timeout anywhere** and `setLoading(false)` was not in a guaranteed path.

- **Hard 3s safety timeout (the core fix).** A single `finishLoading()` resolves `loading` exactly once and tears down timers; an unconditional `setTimeout(… , 3000)` forces it no matter what hangs. Worst case resolves with `user=null` → `/login` → offline notice (recoverable); an infinite spinner is not. All three resolve paths (online handler `finally`, `localRestore` `finally`, safety timer) funnel through `finishLoading`.
- **Non-hanging local restore.** `readLocalSession()` races `getSession()` against a 1.2s timeout — a valid (non-expired) token returns from localStorage instantly even in airplane mode; an expired token's hanging refresh loses the race and is treated as "no session". So the restore itself can never hang.
- **Don't trust `onLine` as the only gate.** Offline-at-mount → `localRestore()` immediately; otherwise a **1.2s fallback timer** runs the same `localRestore()` if auth hasn't resolved (covers the iOS `onLine`-lied case + a swallowed `INITIAL_SESSION`). `localRestore` is a no-op once `resolved` and never hits the network, so the normal online path (INITIAL_SESSION wins first) is untouched.
- **Network-failure fallback in `resolveProfile`.** Online, if `getUserRole` throws or returns null, it now falls back to the cached profile (was: `null` → "Access denied"). Success path unchanged; this only activates on failure (per the spec's "trigger offline path on network failure" hint).
- Every offline-path op is wrapped in try/catch with `finishLoading()` in `finally`. Timeline: non-expired token restores by ~1.2s; expired token → notice by ~2.4s; hard backstop at 3s. **Build green** (38 pages). **Not committed.**
