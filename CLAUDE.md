### P0 follow-up ‚Äî offline auth (cold-start) ‚Äî 2026-06-14, NOT committed

Without this, the route cache was unreachable on a force-close‚Üírelaunch-offline: auth init bounced the driver to `/login`, where sign-in (a network call) also failed. Root cause: `AuthContext` resolves the session via `onAuthStateChange` (localStorage, no network) **but** then calls `getUserRole()` (network `/rest/v1/profiles`); offline, an **expired** access token can't be refreshed ‚Üí `INITIAL_SESSION` emits a null session ‚Üí `page.tsx` (`if (!loading && !user) replace('/login')`) redirects. A **non-expired** token instead left `roles = null` ‚Üí "Access denied." `middleware.ts` `getUser()` is NOT the offline cause ‚Äî the SW serves the cached shell and middleware (edge) is bypassed offline; the gate is purely client-side `page.tsx`.

- **`src/lib/authCache.ts`** ‚Äî `writeCachedProfile` / `readCachedProfile`, key **`ptd_profile_<userId>`** (per-user so a shared device can't cross roles; not sensitive ‚Äî roles are already client-readable). Needed because restoring only the session yields `user` but `roles = null` (getUserRole is network) ‚Üí "Access denied"; the cached profile supplies roles offline.
- **`AuthContext` ‚Äî offline-aware profile resolve.** New `resolveProfile(userId, token)`: **online** runs `getUserRole` exactly as before (outcome unchanged) and additionally writes the profile cache on success; **offline** returns `readCachedProfile` with NO network. Replaces the bare `getUserRole` call inside the `onAuthStateChange` handler. **Online behavior is byte-identical in outcome** (only an added cache write).
- **`AuthContext` ‚Äî offline cold-start safety net.** ONLY when `navigator.onLine === false` at mount: independently `getSession()` (local read for a valid token), trust **only a non-expired** session (`expires_at * 1000 > Date.now()` ‚Äî an expired token can't be refreshed offline), restore `user` + cached profile, and always resolve `loading=false` (so a swallowed/null `INITIAL_SESSION` can't hang the app on "Loading‚Ä¶"). Idempotent with the online `onAuthStateChange` path; the online branch is untouched.
- **`LoginScreen` ‚Äî offline notice replaces the form.** `navigator.onLine` state + online/offline listeners (init `true` for SSR, corrected on mount). Offline ‚Üí renders "You're offline / Open the app while connected first ‚Ä¶ unlocks automatically when you reconnect" instead of the form; `handleSubmit` also guards (no misleading "Invalid email or password" on a network failure). Online path unchanged.
- **Known limitation (by spec):** an **expired** access token offline ‚Üí no valid session ‚Üí the LoginScreen notice (we cannot refresh offline). The cache only fully restores within the access-token lifetime. Also, a driver who last logged in **before** this deploy has no `ptd_profile_*` yet ‚Üí offline would show "Access denied" until one online session re-caches it. New key `ptd_profile_<userId>` ‚Äî no collision with existing keys.
- **Build green** (`npx next build`, 38 pages). First cut committed `27751f4`.

#### Hang fix ‚Äî iOS standalone airplane-mode "Loading‚Ä¶" forever (2026-06-14, NOT committed)

The first cut deadlocked on iOS: force-close in airplane mode ‚Üí relaunch ‚Üí black "Loading‚Ä¶" that never resolved. Two compounding causes: (1) **`navigator.onLine` is unreliable on iOS standalone** ‚Äî it can read `true` in airplane mode for a beat after launch, so the offline net (gated on `onLine === false`) never ran; (2) supabase's init **token-refresh hangs with no network timeout**, so `onAuthStateChange` never fired `INITIAL_SESSION` ‚Üí the handler's `finally { setLoading(false) }` never ran. `loading` stayed `true` forever. There was **no timeout anywhere** and `setLoading(false)` was not in a guaranteed path.

- **Hard 3s safety timeout (the core fix).** A single `finishLoading()` resolves `loading` exactly once and tears down timers; an unconditional `setTimeout(‚Ä¶ , 3000)` forces it no matter what hangs. Worst case resolves with `user=null` ‚Üí `/login` ‚Üí offline notice (recoverable); an infinite spinner is not. All three resolve paths (online handler `finally`, `localRestore` `finally`, safety timer) funnel through `finishLoading`.
- **Non-hanging local restore.** `readLocalSession()` races `getSession()` against a 1.2s timeout ‚Äî a valid (non-expired) token returns from localStorage instantly even in airplane mode; an expired token's hanging refresh loses the race and is treated as "no session". So the restore itself can never hang.
- **Don't trust `onLine` as the only gate.** Offline-at-mount ‚Üí `localRestore()` immediately; otherwise a **1.2s fallback timer** runs the same `localRestore()` if auth hasn't resolved (covers the iOS `onLine`-lied case + a swallowed `INITIAL_SESSION`). `localRestore` is a no-op once `resolved` and never hits the network, so the normal online path (INITIAL_SESSION wins first) is untouched.
- **Network-failure fallback in `resolveProfile`.** Online, if `getUserRole` throws or returns null, it now falls back to the cached profile (was: `null` ‚Üí "Access denied"). Success path unchanged; this only activates on failure (per the spec's "trigger offline path on network failure" hint).
- Every offline-path op is wrapped in try/catch with `finishLoading()` in `finally`. Timeline: non-expired token restores by ~1.2s; expired token ‚Üí notice by ~2.4s; hard backstop at 3s. **Build green** (38 pages). **Not committed.**
