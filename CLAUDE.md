## RFID — where this stands. Read this first if you have been away.

The RFID feature replaces the standalone Easy RFID Pro app for the driver's field workflow. It is not live. No driver has it. Nobody is scanning anything in production.

All the RFID code lives on branch `feat/rfid-native-integration` inside this driver app. It has never been merged to this app's main. Drivers are unaffected and stay unaffected until it is deliberately merged. Putting RFID live is one deliberate merge made on purpose after a device test, never a drift.

Built and tested against a fake scanner, not real hardware: offline database, write queue, delivery checkout, pickup return, touch scan. The scan model is status-first: pick the status, then scan items into it. Individual scan grabs the first tag and has a Clear button. Mass scan collects many, then commits. Tags resolve to real item names instantly because the item table lives on the device.

Never tested on a real device: the RFID radio, GPS at scan time, and live writes to Easy RFID Pro and TapGoods.

The separate partytime-rfid project is a workshop for specs and notes. Its main holds documentation only. No driver has it. When notes say "merged to main," they mean that workshop, not the drivers' app.

Open before a device test: confirm the batch write against the Easy RFID sandbox, test GPS permission on the XR2, replace the name-match fallback with the real rfid_to_tapgoods join.

## RFID Session Guardrails — `feat/rfid-native-integration` (read before ANY RFID work)

Session-scoped, binding rules for the native RFID integration build (started 2026-07-09). Violating any of these fails the run:

- **TapGoods is DRY-RUN ONLY this session.** No live TapGoods write may be sent, regardless of confidence. `TAPGOODS_DRY_RUN` defaults to `true` and is not flipped this session. Dry-run payloads are constructed and logged, never posted. Darren performs live TapGoods verification manually afterward against a clearly labeled test order.
- **Easy RFID Pro writes go ONLY to the sandbox host** resolved from `EASY_RFID_BASE_URL` (default `https://sb-easyrfidpro.ptshome.com`). The client refuses to issue a write when the resolved base URL is not the sandbox, unless an explicit `EASY_RFID_ALLOW_PRODUCTION` flag is set — and that flag is not set this session. The resolved host is asserted and logged once at startup so every run states out loud where its writes are going. Never hardcode a host in the client.
- **The RFID module (`src/modules/rfid/`) may not import from driver-app internals.** App data crosses the boundary only through the injected adapters (`StopContextAdapter`, `IdentityAdapter`, `AuthAdapter`, `LocationAdapter`, `NavigationAdapter`, `ThemeAdapter` — defined in `src/modules/rfid/adapters/types.ts`), and vendor systems are reachable only through ports (`TagBackendPort`, `OrderSystemPort`). The driver app imports from the module, never the reverse. Enforced by an automated boundary test — a violation is a red suite, not a code-review note.
- **Unknowns are logged, never guessed.** Any value that cannot be determined from code, the sandbox, or the SDK notes goes to `docs/ASSUMPTIONS.md` in the format: What I needed / What I did instead / How to verify / Risk if wrong. Then the build continues — never stall on an unknown, never silently pick a plausible value.
- **Never mock a passing test.** A test that cannot pass honestly stays red and gets reported. A green suite that lies is worse than a red one.

## Doctrine / Known Landmines (read every session)

- **Equipment return balance is RESERVATION-SCOPED across all delivery/pickup stops — never pair a pickup with a single delivery stop.** The ledger (`src/lib/equipmentReturns/ledger.ts`): `running_balance = SUM(delivery-stop rows) − SUM(COMPLETED-pickup-stop rows, excluding the current stop)` per `(reservation_id, equipment_key)`. Jobs split equipment across stops (tent delivered separately from inflatable; inflatable picked up before the tent) — the last crew must see what's LEFT after earlier pickups, not any one delivery's original count. The original 2026-07-02 pairwise `linked_stop_id` resolution was **deliberately replaced** the same day — do not revert to it. "Is this the final pickup?" = live query for other incomplete pickups on the reservation (`completed_at IS NULL`); `required_pickup_count` is NOT authoritative (live: 31 multi-pickup reservations vs 23 flagged pickups). The discrepancy alert fires ONLY from the final pickup, once per reservation (`equipment_return_alerts` insert-gate).
- **Inflatable detection is category OR NAME — never category-only — and the Pickup Answer floor MUST equal the early-pickup guard's value.** Category `INFLATABLES` leaks into GAMES / Misc / blank (verified live 2026-07-06: "EMERALD ICE DUAL LANE DRY SLIDE", "CASTLE COMBO -RED/BLUE" sit under a BLANK category), so `src/lib/pickupAnswer/classify.ts` reuses `isInflatableCategory` (from `src/lib/inflatable.ts`, the dashboard mirror) VERBATIM against BOTH category and name — because keyword-less names ("MECHANICAL BULL", "JOUST CHALLENGE") are caught only by their INFLATABLES category, the two gates are OR'd. `src/lib/inflatable.ts` (`hasInflatableItem`) is still category-only by itself — do NOT assume it catches leaked units; the name-aware layer lives in `pickupAnswer/classify.ts`. Tent detection there is a **pure mirror** of `isTentItem` in `ava/dependencyHits.ts` (that module imports the supabase browser client, so it can't be imported into a pure module — keep the two predicates byte-identical). The Pickup Answer card's inflatable "No earlier than" floor is `effectiveWindow(pickupStop).startsAt` — the SAME value the early-pickup standby/countdown guard enforces (`stopConstraints.ts`); never re-derive the pickup time from `scheduled_date` (drifts) or `calculated_eta` (an estimate, not the promise).
- **Chairs are UNRELIABLY categorized in `dispatch_stops.items` — match chairs by item NAME substring, never gate on category.** The same chair name ("CHAIR WHITE PADDED GARDEN", "CHAIR WHITE FOLDING") appears live under **SEATING, Chairs, AND Misc** depending on the order (verified 2026-07-02: CHAIR WHITE FOLDING 247× seating + 27× misc). Any chair-detection rule = lowercase name contains `chair` **minus** names containing `cushion` — "CHAIR CUSHIONS -*" items (filed under LINENS) are a cushion accessory, not a chair, and false-trigger without the exclusion; the exclusion is per-item, so an order with cushions AND real chairs still triggers via the chair line. CHINA / FLATWARE / GLASSWARE categories ARE reliable. Canonical implementation: `EQUIPMENT_RETURN_RULES` in `src/lib/equipmentReturns/rules.ts` — extend that config, don't re-derive the match.
- **`route_assignments` is DROPPED as of Phase 2A** (driver `35e4c37`, 2026-06-09). The table **no longer exists** — a live query returns **PGRST205**. Any reference that destructures only `data` (not `error`) **fails SILENTLY**: `data` → `undefined` → `?? []` → empty set → callers behave as "no driver assigned" with no error. **The correct assignment source is `route_crew`.** Primary-driver lookup: `.eq('is_primary', true).not('user_id','is',null)` (the driver-role value is `'primary_driver'`, NOT `'driver'`; helper/WIW rows carry `user_id = null`). **Always destructure and log `error`** on `route_crew` lookups. Remaining `route_assignments` mentions in `src/` are all comments — verify with `grep -rn route_assignments src/` before adding any assignment-touching endpoint.
- **Realtime subscriptions need BOTH the `supabase_realtime` publication AND the matching RLS SELECT policy — the publication alone is a false green.** Hit twice: `dispatch_stops` (co-driver completion propagation) and `routes` (Route Handoff transfer state). A table can sit in the publication with no readable policy and events will silently never reach the client; always verify both before declaring a realtime feature done.
- **A time-window list filters/groups on the DATE column, never the workflow status — status lags the date.** Will Call's return queue undercounted because it filtered `status === 'awaiting_return'` only; `picked_up` orders due back today were invisible until their status caught up. Any due-date/queue view should key off the actual date column (`checkin_window_end` etc.) with status only driving display treatment.
- **TapGoods write-back 401/403 = an auth/ownership bug, never retried, never queued.** Every other failure mode (network, 5xx, `{synced:false}` retryable reasons) rides the offline queue; a 401/403 means the request was fundamentally wrong and retrying it is pointless — surface it, don't queue it.
- **`stripHtml` (`src/lib/utils.ts`) is required at RENDER time for TapGoods rich-text note fields** (`notes_employee_authored` and similar) — they leak raw `<p>`/`<br>` markup from TapGoods. Apply at render only; the raw value must stay intact in DB/transform for other consumers.

## Architectural invariants — do not re-derive

- **Migration naming (this repo):** `YYYYMMDD<NNN>_*.sql` (concat, NO underscore between date and sequence). Three same-day files → `20260527013/014/015`.
- **`/api/routes` SELECT is the single endpoint for route + stop data.** Never spin up a new endpoint for a `routes`/`dispatch_stops` READ column — add to the existing SELECT + `supabaseTransform` and thread through `SupabaseRouteRow`/`SupabaseStopRow`. (A `routes`-column WRITE gets its own gated endpoint — transfer/depart — this rule governs reads only.) Regen `src/types/supabase.ts` via `supabase gen types typescript --project-id fumprcyavpefyupurvsv` (NOT `--linked` — this repo isn't `supabase link`ed) before changing the SELECT.
- **Field ownership.** `dispatcher_notes` (route + stop) = dashboard-owned, never TapGoods-written. `notes_additional_delivery`, `notes_employee_authored`, `notes_flip`, `notes_set_by_time`, `notes_strike_time` = TapGoods-synced. `ava_stop_notes.note` = driver-authored, address-keyed. `warehouse_notes` (route + stop) = warehouse-owned, wired like `dispatcher_notes`, surfaced BEFORE the dispatcher note (both StopDetail + `StopNotesPreSheet`). `notes_flip` is pickup-only in the pre-launch notes sheet but shows on all stop types in Order Notes.
- **`customerStops`** = `dayStops` minus `warehouse_return`/`warehouse` depot stops (derived once via `useMemo`). ALL counts (stops, COD, tents, checklist manifest, stop-note lookup) route through it.
- **`countTentItems` requires category AND name match:** `category.includes('tent')` AND name contains `tent`/`canopy`/`marquee`. Category-only pulls TENTS-filed "CAFE LIGHTS" (qty 200) into the count.
- **AVA access scoping is server-derived and role-scoped.** `isElevatedRole(roles)` (`src/lib/ava/access.ts`) — "elevated" = `super_admin` (the only elevated role; there is NO plain `admin` role in `profiles.roles`, accepted defensively). Always read the role server-side from `auth.uid()` → `profiles.roles`; never trust a client flag. Default to the driver (most-restrictive) scope if the lookup fails.
- **`/api/ava/ask` system prompt is TWO blocks for caching.** Block 0 = persona + voice rules + role-scoped SOP/vocab/knowledge base — **stable per role**, carries the `cache_control` breakpoint, shared across drivers (the SOP base alone clears Haiku's 4096-token cache minimum). Block 1 = today's route context — **volatile**, after the breakpoint, never cached. **`driverName` + route data must stay in Block 1** or the prefix goes per-driver and kills the cross-driver cache. Model `claude-haiku-4-5-20251001`, `max_tokens: 500`, no `effort`/`thinking` (Haiku 4.5 rejects both).
- **SOP `department` is free-text/composite**, not `'driver'|'field'|'all'` tokens. Driver-visible = department matches `/\b(drivers?|field|all)\b/i` OR title matches `/\b(tent|canopy|marquee)\b/i`. Shared predicate `isDriverVisibleSop` (`src/lib/ava/sopVisibility.ts`) imported by BOTH `SopSearchSection` (client) and `/api/ava/ask` (server) so the Training list and AVA's knowledge never drift.
- **Card visibility triggers (independent):** `stats_enabled`, `ava_stop_notes` hits > 0, (`checklist_enabled` AND dependency hits > 0), `routeDispatcherNote` present, or `stopsWithDispatchNotes > 0`. Route-level dispatcher note renders in TWO non-overlapping windows — PRE-inspection in the AvaMorningCard "From Dispatch" block (`!inspected`), POST-inspection in a persistent block on Home (`inspected === true` AND `dispatcher_notes` non-empty) — the `inspected` split guarantees no duplicate render.
- **`PATCH /api/profile/ava-preferences`** — admin-client UPDATE scoped to three columns only (`checklist_enabled`, `personality_preference`, `stats_enabled`). **No table-level UPDATE policy on `profiles`** (would let a driver mutate `roles`/`fleet_maintenance_access`/`work_order_technician`). Extend this route's allow-list for future driver-editable fields.
- **`AvaChip`** opens the real `AvaConversationSheet` (state lives inside AvaChip → one change covers all 6 screens); passes `routeId = routes.length === 1 ? routes[0].route_id : null`, `seedContext={{}}` (sheet works context-free). Auto-speak is removed — the brief plays only on the gold "▶ HEAR YOUR MORNING BRIEF" tap (`hasSpokenRef` guard, at most once per card mount). Do NOT rename the "Note from dispatch" label.
- **AVA route context is SERVER-LOADED for both today and preview dates** (supersedes the original client-seeded design). `loadRouteDateContext()` splits DELIVERING vs PICKING UP totals; the client seed survives only as the today fallback when the server load returns nothing. Weather stays client-seeded (live Tomorrow.io) and is appended for today only.

## Critical rules (inline — read every session)

- **Pre-push verification.** `npx next build` (not `npx next lint`) must succeed end-to-end before `git push`. Lint does not run the TypeScript type checker.
- **Build-then-push is one indivisible sequence.** Never build and stop. If a build is green and not pushing immediately, name out loud why.
- **Migrations apply via `supabase db push` or `supabase db query --linked --file <path>`** (bypasses the two-repo history block). Never paste into the Supabase SQL Editor. Mark applied with `supabase migration repair --status applied <version>`.
- **Cross-repo helpers are byte-for-byte mirrored.** `src/lib/equipmentSummary.ts`, `src/lib/inflatable.ts`, `src/lib/itemCategories.ts` have twins in `partytime-dashboard`; any change goes to both in the same session.
- **TapGoods API gotchas:** see `docs/claude/doctrine.md` → "Key TapGoods API Learnings".
- **VERSION bump is AUTOMATIC on every driver-facing push.** Any change to `main` touching the driver-facing app (non-API `src/app`, `src/screens`/`components`/`hooks`/`context`, non-type-only `src/lib`, or `public/`) MUST add ONE new entry at the TOP of `RELEASES`/`CHANGELOG` in `src/lib/appVersion.ts` as part of the same push. **Semver:** bugfix-only → PATCH; new feature → MINOR + reset patch; overhaul → MAJOR + reset. **The `.githooks/pre-push` hook is the actual gate** (`scripts/check-version-bump.mjs`; install once per clone via `npm run hooks:install`; bypass `--no-verify`). Non-driver-facing change → `[skip version]` in a commit message. **On a feature branch, `src/modules/` (RFID) is NOT in the guard's driver-facing path list and the guard only runs on pushes to `main`** — so branch commits building RFID need no tag, but do NOT put `[skip version]` on them anyway (see the RFID Merge note above — it poisons the eventual merge-range check).
