## Doctrine / Known Landmines (read every session)

- **A stop has THREE names and AVA must know all of them — never feed an LLM `customer_name` alone.** `dispatch_stops.company_name` is the TapGoods **order name** and is what every driver-facing screen renders (`company_name?.trim() || customer_name` — `DayRouteSelectorScreen`, `StopDetailScreen`, `RouteListScreen`); `customer_name` is the **on-site contact**; `client_company` is the **org**. Live: `company_name` "Camp Kinder Ring - 7/12/2026" / `customer_name` "Gabby x" / `client_company` "CAMP KINDER RING". **407 of 958** live delivery/pickup stops have a `client_company` whose text appears nowhere in `customer_name`. `/api/routes` selects all three; any new SELECT against `dispatch_stops` that feeds a UI or a prompt must too. Canonical name resolution: `stopNames()` in `src/lib/ava/routeContext.ts` — it strips only a **trailing** ` - M/D/YYYY` from the display name; a date **mid-string** is load-bearing (route 3's "NYACK CAMP- JULY 10TH, 9AM-12:30PM" vs "NYACK CAMP- 7/11/2026, …, FAMILY DAY" are different stops for the same customer).
- **AVA's route block numbers stops `Stop N of M` in ONE drive-order sequence — never split deliveries and pickups into separate lists.** A category-split list makes ordinals confidently wrong: on a `del,del,del,del,PICKUP,del` route the 5th delivery is the 6th stop, and when both stops share a `customer_name` the wrong answer reads as correct. Direction lives as a per-row annotation (`· PICKUP`) plus the separate `DELIVERING:` / `PICKING UP:` aggregate totals (the 2026-06-28 fix — preserve those). **The `UNKNOWN:` gap signal must NEVER fire on a route/stop/customer/equipment question** — that data is in Block 1, below the instruction; firing it shows the driver "I'm logging your question" and writes a false `ava_knowledge_gaps` row. **`routeId` from the client SCOPES the context, it never authorizes** — re-validate it against the caller's own `route_crew` rows; null (multi-route day from Home) renders every route, each numbered from 1.
- **Equipment return balance is RESERVATION-SCOPED across all delivery/pickup stops — never pair a pickup with a single delivery stop.** The ledger (`src/lib/equipmentReturns/ledger.ts`): `running_balance = SUM(delivery-stop rows) − SUM(COMPLETED-pickup-stop rows, excluding the current stop)` per `(reservation_id, equipment_key)`. Jobs split equipment across stops (tent delivered separately from inflatable; inflatable picked up before the tent) — the last crew must see what's LEFT after earlier pickups, not any one delivery's original count. The original 2026-07-02 pairwise `linked_stop_id` resolution was **deliberately replaced** the same day — do not revert to it. "Is this the final pickup?" = live query for other incomplete pickups on the reservation (`completed_at IS NULL`); `required_pickup_count` is NOT authoritative (live: 31 multi-pickup reservations vs 23 flagged pickups). The discrepancy alert fires ONLY from the final pickup, once per reservation (`equipment_return_alerts` insert-gate).
- **Inflatable detection is category OR NAME — never category-only — and the Pickup Answer floor MUST equal the early-pickup guard's value.** Category `INFLATABLES` leaks into GAMES / Misc / blank (verified live 2026-07-06: "EMERALD ICE DUAL LANE DRY SLIDE", "CASTLE COMBO -RED/BLUE" sit under a BLANK category), so `src/lib/pickupAnswer/classify.ts` reuses `isInflatableCategory` (from `src/lib/inflatable.ts`, the dashboard mirror) VERBATIM against BOTH category and name — keyword-less names ("MECHANICAL BULL", "JOUST CHALLENGE") are caught only by their INFLATABLES category, so the two gates are OR'd. `src/lib/inflatable.ts` (`hasInflatableItem`) is still category-only by itself — do NOT assume it catches leaked units; the name-aware layer lives in `pickupAnswer/classify.ts`. Tent detection there is a **pure mirror** of `isTentItem` in `ava/dependencyHits.ts` (that module imports the supabase browser client, so it can't be imported into a pure module — keep the two predicates byte-identical). The Pickup Answer card's inflatable "No earlier than" floor is `effectiveWindow(pickupStop).startsAt` — the SAME value the early-pickup standby/countdown guard enforces (`stopConstraints.ts`); never re-derive the pickup time from `scheduled_date` (drifts) or `calculated_eta` (an estimate, not the promise).
- **Chairs are UNRELIABLY categorized in `dispatch_stops.items` — match chairs by item NAME substring, never gate on category.** The same chair name ("CHAIR WHITE PADDED GARDEN", "CHAIR WHITE FOLDING") appears live under **SEATING, Chairs, AND Misc** depending on the order (verified 2026-07-02: CHAIR WHITE FOLDING 247× seating + 27× misc). Any chair-detection rule = lowercase name contains `chair` **minus** names containing `cushion` — "CHAIR CUSHIONS -*" items (filed under LINENS) are a cushion accessory, not a chair; the exclusion is per-item, so an order with cushions AND real chairs still triggers via the chair line. CHINA / FLATWARE / GLASSWARE categories ARE reliable. Canonical implementation: `EQUIPMENT_RETURN_RULES` in `src/lib/equipmentReturns/rules.ts` — extend that config, don't re-derive the match.
- **`route_assignments` is DROPPED as of Phase 2A** (driver `35e4c37`, 2026-06-09). The table **no longer exists** — a live query returns **PGRST205**. Any reference that destructures only `data` (not `error`) **fails SILENTLY**: `data` → `undefined` → `?? []` → empty set → callers behave as "no driver assigned" with no error. **The correct assignment source is `route_crew`.** Primary-driver lookup: `.eq('is_primary', true).not('user_id','is',null)` (the driver-role value is `'primary_driver'`, NOT `'driver'`; helper/WIW rows carry `user_id = null`). **Always destructure and log `error`** on `route_crew` lookups. Remaining `route_assignments` mentions in `src/` are all comments — verify with `grep -rn route_assignments src/` before adding any assignment-touching endpoint.
- **Manifest bundle grouping in the check-off panel keys off `line.index`, never display order.** `ItemCheckoffPanel` rows are built from **`lines`, never `items`** — each row carries its original `line.index` because accept / short-count / damage / commit all key off it. Group by first appearance carrying that index; NEVER re-index rows to their display order or a short-count lands on the wrong item.

## Architectural invariants — do not re-derive

- **Migration naming (this repo):** `YYYYMMDD<NNN>_*.sql` (concat, NO underscore between date and sequence). Three same-day files → `20260527013/014/015`.
- **`/api/routes` SELECT is the single endpoint for route + stop data.** Never spin up a new endpoint for a `routes`/`dispatch_stops` READ column — add to the existing SELECT + `supabaseTransform` and thread through `SupabaseRouteRow`/`SupabaseStopRow`. (A `routes`-column WRITE gets its own gated endpoint — the transfer + depart routes — this rule governs reads only.) Regen `src/types/supabase.ts` via `supabase gen types typescript --project-id fumprcyavpefyupurvsv` (NOT `--linked` — this repo isn't `supabase link`ed) before changing the SELECT. ⚠ A full regen pulls in unrelated RFID tables added to the shared DB — the `profiles` block is hand-patched, not full-regenerated.
- **Field ownership.** `dispatcher_notes` (route + stop) = dashboard-owned, never TapGoods-written. `notes_additional_delivery`, `notes_employee_authored`, `notes_flip`, `notes_set_by_time`, `notes_strike_time` = TapGoods-synced. `ava_stop_notes.note` = driver-authored, address-keyed. `warehouse_notes` (route + stop) = warehouse-owned (wired driver-side like `dispatcher_notes`; surfaced BEFORE the dispatcher note on both StopDetail + `StopNotesPreSheet`). `notes_flip` is pickup-only in the pre-launch notes sheet but shows on all stop types in Order Notes — don't conflate.
- **`customerStops`** = `dayStops` minus `warehouse_return`/`warehouse` depot stops (derived once via `useMemo`). ALL counts (stops, COD, tents, checklist manifest, stop-note lookup) route through it.
- **`countTentItems` requires category AND name match:** `category.includes('tent')` AND name contains `tent`/`canopy`/`marquee`. Category-only pulls TENTS-filed "CAFE LIGHTS" (qty 200) into the count.
- **AVA access scoping is server-derived and role-scoped.** `isElevatedRole(roles)` (`src/lib/ava/access.ts`) — "elevated" = `super_admin` (the only elevated role; there is NO plain `admin` role in `profiles.roles`, accepted defensively). Always read the role server-side from `auth.uid()` → `profiles.roles`; never trust a client flag. Default to the driver (most-restrictive) scope if the lookup fails.
- **`/api/ava/ask` system prompt is TWO blocks for caching.** Block 0 = persona + voice rules + role-scoped SOP/vocab/knowledge base — **stable per role**, carries the `cache_control` breakpoint, shared across drivers (the SOP base alone clears Haiku's 4096-token cache minimum). Block 1 = today's route context — **volatile**, after the breakpoint, never cached. **`driverName` + route data must stay in Block 1** or the prefix goes per-driver and kills the cross-driver cache. Model `claude-haiku-4-5-20251001`, `max_tokens: 500`, no `effort`/`thinking` (Haiku 4.5 rejects both).
- **SOP `department` is free-text/composite**, not `'driver'|'field'|'all'` tokens. Driver-visible = department matches `/\b(drivers?|field|all)\b/i` OR title matches `/\b(tent|canopy|marquee)\b/i`. Shared predicate `isDriverVisibleSop` (`src/lib/ava/sopVisibility.ts`) imported by BOTH `SopSearchSection` (client) and `/api/ava/ask` (server) so the Training list and AVA's knowledge never drift.
- **Card visibility triggers (independent):** `stats_enabled`, `ava_stop_notes` hits > 0, (`checklist_enabled` AND dependency hits > 0), `routeDispatcherNote` present, or `stopsWithDispatchNotes > 0`. Route-level dispatcher note renders in TWO non-overlapping windows — PRE-inspection in the AvaMorningCard "From Dispatch" block (`!inspected`), POST-inspection in a persistent block on Home (`inspected === true` AND `dispatcher_notes` non-empty) — the `inspected` split guarantees no duplicate render.
- **`PATCH /api/profile/ava-preferences`** — admin-client UPDATE scoped to three columns only (`checklist_enabled`, `personality_preference`, `stats_enabled`). **No table-level UPDATE policy on `profiles`** (would let a driver mutate `roles`/`fleet_maintenance_access`/`work_order_technician`). Extend this route's allow-list for future driver-editable fields.
- **`AvaChip`** opens the real `AvaConversationSheet` (state lives inside AvaChip → one change covers all 6 screens); passes `routeId = routes.length === 1 ? routes[0].route_id : null`, `seedContext={{}}` (sheet works context-free). Auto-speak is removed — the brief plays only on the gold "▶ HEAR YOUR MORNING BRIEF" tap (`hasSpokenRef` guard, at most once per card mount). Do NOT rename the "Note from dispatch" label.

## Subsystem reference (operative rules — history in `docs/CHANGELOG.md`)

### Equipment Returns (migs 028/029)
Shared rules config `src/lib/equipmentReturns/rules.ts` is the ONE file both sides import + the integrity source (table deliberately has NO key CHECK — new types are a code change). Triggers: `extension_cords`=always; `china_racks`/`glassware_racks`/`flatware_crates`=category; `chair_carts`=`name_contains 'chair'` minus `cushion`, category ignored, ONE stepper regardless of chair style. Delivery capture = `EquipmentReturnSection` (soft prompt, never gates completion; touched steppers upsert via `commitEquipmentReturns` in `runStopComplete`, queue `ptd_equipreturn_queue` + draft `ptd_equipreturn_draft:<stopId>`). Pickup surfacing = `GET /api/stops/equipment-returns?stop_id=` (service-role + cookie, always `200` on no-data/error — linked delivery is on a DIFFERENT route so client RLS can't read it) → `EquipmentRetrieveCard`. Ledger revision: GET returns `returns` (cards) + `balances` (prefill); POST = pickup-completion path (crew-gated, live incomplete-pickups check decides finality, `retryAlert` keeps the client queue retrying). Alert email = `alertEmail.ts` (Resend HTTP, from `alerts@partytime-rentals.com` / to `dispatch@partytime-rentals.com` — the HYPHENATED domain is Resend-verified; un-hyphenated 403s; `RESEND_API_KEY` set + verified). `EquipmentPickupSection` pre-fills `balance`; unconfirmed rows write NOTHING; below-prefill entries get a non-blocking "fewer than expected" note.

### Pickup Answer (driver-facing, no mig, v2.5.0)
Gold card on the delivery stop, `src/components/pickup/PickupAnswerCard.tsx` (self-contained, delivery-only, under the customer card). Pure derivation `src/lib/pickupAnswer/derive.ts` (+ `types`/`classify`/`format`): group by `reservation_id` (NOT `linked_stop_id` which chains pickup→pickup), dedup phantom rows by `scheduled_date`. Endpoint `GET /api/stops/pickup-answer?stop_id=` (service-role + cookie, `runtime=nodejs`). **The `ok` discriminator is load-bearing:** "no pickup scheduled" copy shows ONLY on `ok:true, kind:'none'` — a transient failure returns `ok:false` → card renders nothing (never tells the customer "no pickup" on a failed load). All time formatting via `Intl` America/New_York (DST-safe). Split labels ("Inflatable pickup"/"Tent pickup") only when exactly two distinct dates AND cleanly separable, else neutral "First/Second trip". See Known Landmines #4 for the guard-alignment invariant.

### DOT Inspection Safety Net — truckless-primary guard (no mig)
The pre-trip gate keys on the driver's OWN `route_crew.truck_id` (`Route.truck_is_own`), NOT the route-level trucks — **correct, do not "fix" it.** A lone primary whose crew row has a null `truck_id` gets `truck=null`, `truck_is_own=undefined` → `stopsLocked` false → bypasses the pre-trip. Fix = additive guard `truckUnassigned = !hasTruck && is_primary === true` in BOTH `DayRouteSelectorScreen` + `RouteListScreen` (fix-both mirror; the inspection gate is UNTOUCHED): disabled CTA + "Contact dispatch to assign your truck", and stops non-tappable via `stopsTappable = !stopsLocked && !truckUnassigned` (closes the Routes-tab deep-link bypass). Scoped to `is_primary === true` only — co-drivers and soft-fail no-crew-row are not caught.

### Stop progression gate + optimistic completion + auto-ETA (dashboard mig 097)
**Mig 097 = `profiles.auto_send_eta BOOLEAN NOT NULL DEFAULT FALSE`** (admin-set only; applied from the dashboard mirror). Part 1: `runStopComplete` calls `markComplete` (local flip) BEFORE the `/api/complete-stop` POST → driver advances offline; failures queue in `src/lib/completeQueue.ts` (`ptd_complete_queue`, dedupe-by-stop, flushed in `loadDay`); endpoint idempotent. Part 2: `isProgressionLocked(stop)` per-stop gate in both selector screens (fix-both) — depot legs + completed never lock; the first not-yet-completed customer stop **on that route** is unlocked, later stops locked; **route-scoped `.find`**, layered ON TOP of the inspection `stopsLocked` gate; dimming is the only cue. Part 3: `maybeFireAutoEta()` in `runStopComplete` — if `readCachedProfile(uid)?.auto_send_eta`, sends next-customer OTW via `sendEtaSms`; fire-and-forget, self-gates on `navigator.onLine` + GPS + a phone, completely invisible; offline → silent skip. `loadDay` refreshes `ptd_profile_<uid>` each online load so an admin toggle lands without restart (**no `/api/profile` route exists**). Dashboard side: `PATCH /api/admin/drivers/[id]/preferences` (super_admin, service-role write).

### TapGoods Item Check-Off (dashboard mig 096)
Inline `ItemCheckoffPanel` renders in the manifest slot when `checkoffActive` (delivery/pickup + items + not completed + not committed + `canComplete`); a single gated bottom CTA does completion (funnel: **items → cash → complete**). **Available to ALL crew with completion rights — never primary-only** (the only primary-only action is ETA SMS). Committed state = localStorage `ptd_checkoff_done:<stopId>` OR any existing `stop_item_checkoffs` row (RLS crew_read probe); draft in `ptd_checkoff_draft:<stopId>`. Audit rows are **append-only** (no driver UPDATE/DELETE; re-confirm = new row, latest `confirmed_at` wins) — so a damage flow's `work_order_id` MUST be captured BEFORE commit and ride the INSERT. Write-back: `POST {NEXT_PUBLIC_DASHBOARD_URL}/api/tapgoods/dispatch/write-back`, bearer auth, body `{stop_id, lines:[{tapgoods_pick_list_item_id, qty}]}` (server derives ordered qtys + Option A/B shape, emails once per stop, maps delivery→`in_use`/pickup→`checked_in`). **401/403 = auth/ownership bug — never retried, never queued.** Offline queue `ptd_checkoff_queue`: enqueue on network failure + retryable `{synced:false}` reasons; drop provably-permanent reasons (`bad_request`/`stop_not_found`/`unsupported_stop_type`/`no_tapgoods_ids`); `no_rental_id` retries. Damage → WO via `report-issue?…&checkoff=1` (row lands `damaged=true, work_order_id` = the WO or null if skipped). Silent dashboard accessory/add-on sweep writes them to FULL qty, NEVER into `dispatch_stops.items` or any UI.

### Route Handoff — Phase 2B (dashboard mig 093)
`routes.active_driver_id` (NULL ⇒ `is_primary` controls; set ⇒ that profile owns SMS/ETA/completion + can re-transfer) + `routes.transfer_pending_to`. State machine: Idle → Pending → Transferred **or** Declined; **no reclaim**. Identity: `profiles.id` = `auth.users.id` = `route_crew.user_id` (compare directly to `AuthContext` id). Ownership primitive `src/lib/routeOwnership.ts` `isActiveDriver(route, profileId)` — **use for ALL SMS/ETA/completion gates, never read `is_primary` raw again.** Completion is ACTIVE-TRANSFER-ONLY: `canComplete = !route.active_driver_id || isActiveDriver(...)` (no transfer → anyone completes, incl. co-drivers; transfer active → only the active driver). Endpoints (driver): `POST /api/routes/transfer/initiate` (owner-gated) + `/respond` (recipient-gated) — cookie auth identifies caller, service-role does the writes, gates 100% server-side. `/api/complete-stop` clears both transfer columns on a `warehouse_return` completion (route end). Realtime: `routes` needs BOTH the `supabase_realtime` publication AND the `routes_authenticated_read` RLS policy — publication alone is a false green.

### Will Call — Phase 1 (no mig)
The role is **`will_call`** (`will_call_board` is only a realtime channel name — never a role). All four dashboard `/api/willcall/[id]/*` routes are gated `requireWillCallAccess` = cookie → Bearer (cross-origin) → role gate `will_call|warehouse|scheduler|super_admin`; CORS + OPTIONS for `work.partytime-rentals.com` (a bearer retrofit for a browser caller is incomplete without CORS). Driver reads: `GET /api/will-call` (same four-role gate, admin-client SELECT, non-returned + returned-within-7-days). **Writes = cross-app POST only** (`src/lib/willCall/api.ts` → dashboard stage/pickup/return) — never call partytime-sms directly, never write `will_call_orders` via supabase (the dashboard routes own the SMS/email). **No realtime subscription** (postgres_changes are RLS-filtered per subscriber → events silently drop for `will_call`-only holders) — `useWillCallOrders` refetches on focus/visibilitychange + 30s interval. Check-off state is client-side only (Phase 1). `STATE_PILL` labels are LOCKED mockup strings ("Needs Staging"/"Staged — Ready"/"Out w/ Customer"/"Return Overdue"/"Returned ✓"). Due-back date = `checkin_window_end` (NOT `checkin_window_start`), fallback `return_reminder_date`. **Any time-window list filters/groups on the DATE column, not the workflow status** (status lags the date). Nav: `will_call` holders get the Will Call tab in Training's slot; Training moves to a Tools Hub card.

### Warehouse Departure / IN TRANSIT writer (dashboard mig 095)
`routes.actual_departure_at` — stamped when the driver starts the route so the warehouse 5-stage tracker + board `'out'` column advance on departure (`dispatch_stops.actual_departure_at` is the per-stop leg, wrong grain). **`POST /api/routes/[routeId]/depart` is the ONLY writer** (no body; cookie auth + service-role; ownership gate mirrors transfer/initiate — only primary/active driver, not co-drivers; idempotent, `.is('actual_departure_at', null)` guard). `src/lib/departApi.ts departRoute()` is best-effort — never throws, never blocks navigation. Two call sites: InspectionScreen complete-step (stamps unless outcome `oos`) + Home "Start Route" branch (stamps only when `hasTruck && inspected`; "Join Route" does NOT stamp).

### Co-driver realtime completion (no mig)
The subscription lives in `AppStateProvider` (root layout, always mounted), **NOT a screen** (App Router unmounts `DayRouteSelectorScreen` on navigation — the exact bug scenario). Subscribes to `dispatch_stops` UPDATE; on any event whose `route_id` is in today's set → `loadDay(today, true)`. Client-side route-id filter (server-side realtime filters are false-green-prone). Requires BOTH `dispatch_stops` in the `supabase_realtime` publication AND the `dispatch_stops_authenticated_read` SELECT policy.
