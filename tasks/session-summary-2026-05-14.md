## What shipped

### Commit `ff006c6` — driver-scope routes + remove cold-load auto-redirect

**Three concrete code changes, one commit:**

1. **`src/app/api/routes/route.ts`** is now session-aware. Reads the auth cookie, joins `route_assignments` for the requested date, narrows the routes response to only the caller's assignments. Falls back to all routes when no assignment exists or the caller has no session (preserves dispatcher/admin tooling). Cache-Control flipped from `public, max-age=30, stale-while-revalidate=15` to `private, no-store` because responses now vary per user. Super_admin gets the same scoping as driver — driver app is the driver's tool, dashboard is the god view.

2. **`src/hooks/useAssignedRoute.ts`** deleted.

3. **`src/app/api/routes/assigned/route.ts`** deleted (orphan after hook removal).

4. **`src/screens/DayRouteSelectorScreen.tsx`** had its `useAssignedRoute` import, the `showAssignmentLoader` branch in the scroll body, and the assignment-spinner UI removed. Cold sign-in now lands on Home, always.

5. **Comment refs** in `src/app/api/routes/route.ts` and `src/app/api/schedule/week/route.ts` updated to reference `/api/inspection/status` instead of the deleted `/api/routes/assigned`.

This fixed Bugs 1, 2, and 3 with one architectural change: scope the data, remove the redirect.

### Commit `c308c81` — regen Supabase types post-migration-051

`cash_collections.status` and `not_collected_reason` added to the autogenerated types. Cleaned up two CLI stderr lines (`Initialising login role…` header, `A new version…` footer) that the gen-types redirect leaks into the file.

### Commit `4788034` — persist completion across refetches

Bug 4. **Three changes in three files**, no `StopDetailScreen` JSX touched:

1. **`src/app/api/routes/route.ts`** — added `stop_status, completed_at` to the `dispatch_stops` SELECT (previously omitted — server wrote completion but client never re-read it).

2. **`src/lib/supabaseTransform.ts`** — extended `SupabaseStopRow` with `stop_status` and `completed_at`. `toRealStop` now maps `stop_status === 'completed'` → `current_status: 'completed'` and carries `completed_at` through. Default `'pending'` otherwise.

3. **`src/context/AppStateContext.tsx`** — `loadDay`'s merge short-circuits on server-completed: `if (s.current_status === 'completed') return s` before the OTW / localStorage lookups. Completion is terminal; OTW state from `StopStateService.readOtwStatus` can no longer clobber it. The 5s `setTimeout(loadDay(date, true))` in `runStopComplete` is untouched — the dashboard ETA cascade still needs that window to write back fresh `calculated_eta`.

The `StopDetailScreen`'s `{isCompleted ? <Delivered> : <ETA + Action card>}` ternary at line 1163 was already correct; `isCompleted` just couldn't ever evaluate true on return navigation because the data was wrong. **Symptom in the JSX, root cause in the data layer.** Added as a lesson.
