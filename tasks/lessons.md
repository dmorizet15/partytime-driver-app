# Lessons — partytime-driver-app

Patterns to avoid and reinforce. Read at session start. Append when Darren corrects something or when a non-obvious pattern proves itself out.

Format: one lesson per block. Lead with the rule, then **Why** and **How to apply**.

---

## `supabase db query --linked --file <multi-statement.sql>` returns ONLY the last statement's result set. Intermediate `SELECT`s (counts, smoke checks) in the same file are run but their output is discarded — re-run them as single-statement queries to actually see the numbers.

**Why:** 2026-06-21 the AVA inflatable specs patch ended with three verification `SELECT`s (an UPDATE-count check, a `COUNT(*)`, and a Wild Rapids spot-check, plus an ordered listing). The CLI echoed only the **final** statement (the ordered listing of all 57 rows); the `pirate_battle_rows_updated` count and the `total_inflatable_entries` count never appeared. The statements DID execute against the DB — the CLI just serializes one result set per invocation. Confirming the expected counts required a second, single-`SELECT` query.

**How to apply:** when you need to *report* a count/verification from `db query --linked --file`, put it in its OWN file as the sole (or last) statement, or run a separate one-liner afterward (read-only `SELECT`s are always safe to re-run). Combine the checks you need into one final `SELECT` with sub-selects (`SELECT (SELECT COUNT(*) …) AS a, (SELECT COUNT(*) …) AS b;`) so a single statement returns everything. Don't assume a silent/missing result means the statement failed — it ran; you just didn't get to see it.

## `ava_knowledge.question` has NO UNIQUE constraint — an INSERT-based data patch into it is NOT idempotent. Run it exactly once; a second run silently inserts duplicates. Prefer UPDATE + guarded INSERT for re-runnable patches, and when that's not practical, label the file "DO NOT RE-RUN" loudly.

**Why:** 2026-06-21 the inflatable specs patch was 1 UPDATE + 56 plain INSERTs. With no unique key on `question`, re-running it would have added 56 duplicate rows that AVA would then inject into its knowledge base (no error, no signal). This is unlike the `dependency_map` data-patches under `supabase/data-patches/`, which use `UPDATE … WHERE` + `INSERT … WHERE NOT EXISTS`-style guards and are safe to replay (and self-heal a fresh DB rebuild that restored the migration-seed defaults). The inflatable patch can't self-heal the same way — on a rebuild it must be applied once and only once.

## When a data-patch family splits a base file (INSERTs) from a fixup file (UPDATEs a row the base inserts), run order is strict — base first. Running the fixup first makes its `UPDATE … WHERE question = …` match 0 rows and silently no-op; the count comes up short and the correction is lost. If the base file is missing, flag the dependency and HOLD the dependent run — don't no-op into a state that's hard to clean.

**Why:** 2026-06-21, the tent specs went in as families with a base + a fixup. `ava_stillwater_44_weights_v2.sql` (5 UPDATEs + 5 INSERTs) was run before `ava_stillwater_44_weights_total.sql` existed → the 5 UPDATEs hit 0 rows (the total-weight questions weren't in the DB yet), count landed 20 not 25. Same shape for 30-wide: `ava_30wide_frame_spreaders.sql`'s UPDATE targets `What goes on the 30 by 45 frame tent?`, a row created by `ava_30wide_frame_tent.sql` — and that base file was *missing from disk* (referenced at a `/mnt/user-data/outputs/` path that doesn't exist on this Mac). Running spreaders first would have: (a) reported 27 not 35, (b) no-op'd the "11 spreaders" fix, and (c) — because there's no UNIQUE on `question` — made re-running spreaders later (to re-apply the fix once the base arrived with the wrong "6 spreaders" text) duplicate the 2 spreader rows. Unlike the earlier weights case where the base carried the *final correct* text (making the fixup moot), here the base carries *wrong* text the fixup corrects, so base-then-fixup is mandatory and not self-correcting.

**How to apply:** before running a `*_v2`/`*_fixup`/`*_spreaders`-style patch, confirm the base rows it UPDATEs actually exist (`SELECT … WHERE question = '…'`). If they don't — or the base `.sql` is missing — stop and ask for the base rather than running the fixup into a 0-row no-op. Record the run order in the patch header + CLAUDE.md (Stillwater = specs → weights_total → weights_v2; 30-wide = frame_tent → spreaders). And don't trust file paths in a prompt: `/mnt/user-data/outputs/…` is not a path on this Mac — search the filesystem / `~/Downloads` / `supabase/data-patches/` first, and if the file isn't there, say so instead of assuming it exists.

**How to apply:** before writing an INSERT data-patch, check whether the target column/table has a UNIQUE/PK that makes a re-run a no-op (or an ON CONFLICT target). If not: (1) gate each INSERT with `WHERE NOT EXISTS (SELECT 1 … WHERE question = …)` to make it replay-safe, OR (2) accept one-shot semantics and put a bold "DO NOT RE-RUN — no UNIQUE on question, duplicates insert silently" header in the `.sql` so future-you / a rebuild script applies it once. Either way, state the idempotency property in the CHANGELOG entry. Data patches live forever and get re-applied on rebuilds — "run once" must be encoded in the file, not just remembered.

## A table migration (`route_assignments` → `route_crew`, Phase 2A) was not applied uniformly — two stale references survived for weeks, hidden by silently-swallowed query errors. Any new endpoint touching driver↔route assignment MUST use `route_crew`, and MUST destructure `error`, not just `data`.

**Why:** 2026-06-09 Phase 2A (`35e4c37`) migrated driver assignment from `route_assignments` to `route_crew` and **the old table was dropped from the DB**. `/api/routes` was updated, but two readers were missed: `src/app/api/schedule/week/route.ts` (fed the Week View "YOU" badge's `driver_id`) and `src/lib/personalStatsClient.ts`. Both kept querying the dropped table. The week-view query did `const { data: assignments } = await db.from('route_assignments')…` — destructuring ONLY `data`, never `error`. A query against a non-existent table returns PGRST205 in `error` and `null` in `data`; `(assignments ?? [])` then silently became `[]`, so every route serialized with `driver_id: null` and the badge could never match `route.driver_id === currentUserId`. No exception, no 500, no log — the bug was invisible at runtime and survived weeks until someone noticed the badge was gone (fixed 2026-06-17, `3173c48`).

**How to apply:** (1) **When a table is dropped/renamed, grep the WHOLE repo for the old name before shipping** — `grep -rn route_assignments src/` — and migrate every reader in the same change, not just the obvious one. A "scope /api/routes" commit is not the same as "migrate everything off the old table." (2) **Destructure and surface `error` on every supabase query.** `const { data, error } = …; if (error) console.error(...)`. Destructuring only `data` turns a dropped-table / RLS / schema error into a silent empty result that mimics legitimate "no rows" — the single most effective mask for this class of bug. (3) `route_crew` is the assignment source of truth now; the driver-role value there is `'primary_driver'` (not `'driver'`), helper/WIW rows have `user_id = null`, so a primary lookup is `.eq('is_primary', true).not('user_id','is',null)`. (4) See CLAUDE.md → "Doctrine / Known Landmines" for the standing rule.

## A gate that "looks driver-count-dependent" is usually keying on a per-driver field, not a count. The DOT inspection gate keys on the signed-in driver's OWN `route_crew.truck_id` (`truck_is_own`), never the route-level trucks (`routes.truck_id`/`truck_id_2`) — and when a crew row exists with a NULL truck, the app silently falls through to "Join Route" and bypasses the pre-trip.

**Why:** 2026-06-16 the DOT inspection didn't show for a route with 1 driver + 2 trucks but did once a 2nd driver was added (trucks unchanged). The trucks ON the route were never the gating input. `supabaseTransform.ts` resolves the driver's truck from THEIR crew row: `ownTruck = mine ? firstRel(mine.truck) : null`, then `truck = mine ? ownTruck : firstRel(r.truck)`. The route-truck fallback (`firstRel(r.truck)`) fires ONLY when there's no crew row at all (`mine` null → soft-fail) — NOT when a crew row exists with a null `truck_id`. So a lone primary whose crew row was never assigned a truck gets `truck = null` → `hasTruck = false`, `truck_is_own = undefined` → `stopsLocked` (needs `truck_is_own === true`) is false → stop list never locks AND the CTA reads "Join Route" (`!hasTruck`) → tapping routes straight past the DOT pre-trip. Adding a 2nd driver made dispatch pin one truck per crew row, so each driver's `route_crew.truck_id` populated and the gate fired — the "driver-count" symptom was really "was a truck assigned to my crew row."

**How to apply:** when a gate's behavior tracks some count (drivers, stops, items), trace it to the actual field it reads before assuming the count matters — here the chain was `stopsLocked → truck_is_own → ownTruck → route_crew.truck_id`. Two takeaways for fixes: (1) **a missing per-driver assignment must never be a SILENT pass on a regulatory gate** — guard it and surface the reason (a disabled CTA + "contact dispatch" beats falling through to a different flow). (2) When adding such a guard, **scope it to the role that's actually expected to have the resource** (`is_primary === true` for the truck/inspection case) so you don't catch supported edge users — a no-truck ride-along co-driver (`is_primary === false`) is legitimately truckless and must keep its normal flow. (3) `stopsLocked` does NOT cover the truckless-primary case (`truck_is_own` is `undefined`, not `true`), so a deep-linkable surface like RouteListScreen needs the guard folded into `stopsTappable` separately — additive, not a change to the inspection gate. The root data fix (dispatch pinning a truck to a lone crew row on a multi-truck route) lives in the dashboard repo; the driver app gets the guardrail.

## RLS is ROW-level, not COLUMN-level. To let "anyone update only columns X and Y" while authors keep full edit rights, pair a permissive UPDATE policy with a `BEFORE UPDATE` trigger that rejects foreign edits to other columns — and make the trigger skip when `auth.uid()` IS NULL so the service role still works.

**Why:** 2026-06-15 (Ava Studio A4) the freshness-confirm needed ANY visiting driver to mark an `ava_stop_notes` row "still good" (`last_confirmed_at` + `visit_count_since_added`), but NOT to rewrite another driver's note text. The instinct is "add an UPDATE policy." But RLS policies gate ROWS, not columns — and multiple permissive policies are OR'd, so a bare `FOR UPDATE TO authenticated USING(true) WITH CHECK(true)` would have OVERRIDDEN the existing author-only `update_own` and let anyone edit anything. Postgres has no per-column RLS; column privileges (`GRANT UPDATE(col)`) aren't row/author-conditional, so they can't express "author → all columns, others → 2 columns" either. The clean expression is RLS-for-rows + a trigger-for-columns: a permissive policy opens the row to any authenticated user, and a `BEFORE UPDATE` trigger raises if a non-author (`auth.uid() IS DISTINCT FROM OLD.author_id`) changed anything beyond the two freshness columns. The critical gotcha: the **service role** (used by the archive route) runs with `auth.uid() = NULL`, and `NULL IS DISTINCT FROM <uuid>` is TRUE — so without an `auth.uid() IS NOT NULL` guard the trigger would block the service-role archive (which legitimately flips `status`). Lead the trigger with `IF auth.uid() IS NOT NULL AND auth.uid() IS DISTINCT FROM OLD.author_id THEN …`.

**How to apply:** when a spec says "any user can update only these fields," reach for policy+trigger, not a policy alone. Permissive UPDATE policy (`USING(<row scope>) WITH CHECK(true)`), then a `BEFORE UPDATE` trigger comparing `NEW.<col> IS DISTINCT FROM OLD.<col>` for every column the foreign user must NOT touch, raising on any mismatch. Always exempt the no-JWT path (`auth.uid() IS NULL`) so service-role writes pass. And remember the management API (`db query --linked`) runs WITHOUT a JWT — so it CANNOT test the cross-author RLS path (it bypasses RLS as superuser); that verification is on-device only. Don't mark an RLS feature "done" off a management-API smoke.

## A spec written as a "voice command" / "read aloud" / "AVA reads it" may assume capabilities the app doesn't have. Verify the actual I/O surface before building — `AvaConversationSheet` has NO speech-to-text (mic is a stub), so any "voice command" must degrade to a typed/button trigger.

**Why:** 2026-06-15 (Ava Studio A4) the locked spec's "Start Fresh **voice** command" said: in `AvaConversationSheet`, detect intent "start fresh"/"clear notes" for the current address. Two facts the spec didn't account for: (1) the sheet is **text input only** — line 9 of the file literally says "voice input is future scope"; the mic button is a UI stub. There is no STT to detect a spoken command from. (2) The sheet isn't address-scoped — it's opened globally from the header chip with an empty `seedContext`, so it has no "current address" to act on. Building the spec verbatim would have meant retrofitting address-context threading through the chip just to make a *typed* "start fresh" string work — a lot of plumbing for a phantom capability. Darren's call: ship it as a plain "Clear site notes" **button** on Stop Detail (where the address is already in scope), confirm dialog → archive route; leave the voice command in the spec for when mic input is real.

**How to apply:** when a spec leans on an interaction modality (voice, camera, push, biometrics), grep the relevant component for the actual capability before designing around it — don't assume the modality exists because the spec uses it. If it's a stub, surface that as a decision (degrade to the nearest real affordance: button for voice, manual entry for camera) rather than silently building the verbatim-but-broken version. Same family as "verify the role-column shape against the live DB, not the spec." A locked spec is intent, not ground truth about the codebase.

## When a UI element is rendered per-screen (not in the layout), wire its shared state INSIDE the component, and pull "available" context from the global store (`useAppState`) — not by prop-drilling through every screen.

**Why:** 2026-06-15 (AVA header chip activation) the spec said "open the sheet from any screen" and assumed the chip lived at layout level. It does not — `<AvaChip/>` is rendered bare in 6 separate screens (Home, Route list, Stop detail, Tools, Profile, Training); the layout only mounts `AppStateProvider`. Putting `open`/`isAvaOpen` state in the layout would have done nothing for the chip; threading a sheet + context props into all 6 screens would have been a large, error-prone change for a "wiring only" task. Wiring the `open` state + the `<AvaConversationSheet>` render INSIDE `AvaChip` itself made it one file, all screens, zero per-screen edits — exactly mirroring how `AskAvaButton` already self-contains its sheet.

**How to apply:** before reaching for "lift state to the layout," grep where the component actually renders (`grep -rn "<AvaChip" src/`). If it's rendered N times across screens, co-locate the state in the component. For context the sheet/endpoint wants, read it from the global store the component can already reach (`useAppState().routes`) and degrade gracefully when it's ambiguous — e.g. `routeId = routes.length === 1 ? routes[0].route_id : null` (a single route is unambiguous; multiple/none → null + empty `seedContext`). The spec's "pass context where available, never required, no prop drilling, silent degradation" is the right default for any app-global entry point. (Also: `Route.id` does not exist in this repo — the field is `route_id`. Check `src/types/index.ts` before assuming `.id`.)

## When a spec hands you SQL referencing a `profiles` role column, verify the column's name + shape against the LIVE shared DB before applying — never against this repo's local migrations. This DB uses `roles text[]`, not `role`.

**Why:** 2026-06-15 (Ava Studio A1) the locked spec's RLS policies all used `exists (select 1 from profiles where profiles.id = auth.uid() and profiles.role = 'super_admin')`. The driver-app's own local migration `20260426001_create_profiles.sql` still defines `role user_role` (a singular enum) — so the spec looked self-consistent. But this is a TWO-REPO shared DB owned by the dashboard, and `profiles` migrated `role` (enum) → `roles` (text[] array) long ago. Applying the spec verbatim would have failed every migration with Postgres `42703` "column profiles.role does not exist" (PostgREST even hints "Perhaps you meant profiles.roles"). The whole driver-app codebase already proves the truth: the route reads `.select('roles')` + `isElevatedRole(profile.roles)`, and migrations 014 + 016 gate with `'super_admin' = ANY(p.roles)`.

**How to apply:** before applying any role-gated SQL, confirm the live column with a quick probe — `curl "$SUPABASE_URL/rest/v1/profiles?select=roles&limit=1"` with the service key (a wrong column 400s with the 42703 hint), or `supabase db query --linked "select column_name from information_schema.columns where table_name='profiles' and column_name in ('role','roles');"`. The local `supabase/migrations/` are NOT the source of truth for shared dashboard-owned tables — they lag the live schema. The correct super_admin check here is `'super_admin' = ANY(p.roles)` (copy migs 014/016). A deterministic, codebase-proven correction like this is in-scope to fix autonomously (not a "stop and ask"), but flag it loudly in the session summary since it deviates from a locked spec.

## iOS does NOT throw or reject a `fetch()` when the network drops mid-flight — it stalls indefinitely. Any code after `await fetch(...)` (including `setLoading(false)`, `closeModal()`, `setCashSubmitting(false)`) is permanently frozen until iOS's own multi-minute TCP timeout. Add an `AbortController` timeout to EVERY action-critical fetch.

**Why:** 2026-06-22 drivers reported the app freezing when cell reception dropped while completing a stop or confirming cash. The stop-complete spinner never cleared, the cash modal locked with all inputs disabled, and the driver had no way to dismiss or retry. Root cause: on iOS, a fetch that starts while online and then loses the connection does NOT reject — the promise hangs in the pending state for minutes. Unlike desktop browsers or Android where network errors resolve quickly to a `TypeError: Failed to fetch`, iOS's TCP stack keeps the connection alive (SYN retransmit cycle) for a long time before giving up. So `await fetch(url)` just never settles, and the code below it — including all UI-unlock calls — never runs. A simple `navigator.onLine === false` guard only fires when the connection is already known to be down BEFORE the fetch; it does nothing for mid-flight drops.

**How to apply:** every action-critical fetch (one that blocks user-visible UI) needs an `AbortController` timeout:
```ts
const ctrl = new AbortController()
const tid  = setTimeout(() => ctrl.abort(), 10_000)  // 8–12 s depending on criticality
const r = await fetch(url, { …, signal: ctrl.signal }).finally(() => clearTimeout(tid))
```
Put the timeout in `finally` so it clears on BOTH success and abort. Wrap in try/catch:
- **Background/status fetches** (no modal, just derived state): catch → fail closed, log the error. E.g. `useInspectionStatus` → `setInspection(null)` (gate stays closed = safe default).
- **Action-blocking fetches** (completion, cash, OTW): catch → surface a dismissible error message (`setCashError`, `setShowError`) + unlock the UI (`setCashSubmitting(false)`, `setLoading(false)`) so the driver can retry or bail. For completion specifically: the optimistic `markComplete()` already runs BEFORE the fetch, so on timeout route into `enqueueCompletion()` (same path as offline) and navigate forward — the driver is not stranded.
- **Corollary — `visibilitychange` for foreground restoration:** iOS's `window 'online'` event does NOT fire when the app comes from background to foreground if the network state didn't change (was online before, still online after). Add a `document.visibilitychange` listener that calls `loadDay` when `visibilityState === 'visible'` — this is the reliable foreground-restoration trigger. `loadDay` is already abort-controller-guarded (10 s) + has an offline fast-path, so it's safe to call on every foreground event. The `online` event stays for the network-recovery case.

## Apply driver-app-OWNED migrations to the shared DB with `db query --linked --file` + `migration repair`, not `supabase db push`.

**Why:** 2026-06-15 — `db push` compares local migrations against the remote `schema_migrations` tracker and pushes everything unapplied. On this two-repo shared DB the remote tracker holds 90+ dashboard migrations the driver app has never seen, and the driver app's local history (001–021) may not all be recorded remotely → push risks a history-mismatch block or re-running already-applied DDL. The spec said `supabase db push`, but CLAUDE.md's established path (used for migs 093/095/097) sidesteps the whole problem: run the SQL directly via the Management API, then mark it applied.

**How to apply:** for each migration file, `supabase db query --linked --file <path>` (the repo IS linked to `fumprcyavpefyupurvsv`; `--linked` goes through the Management API, no DB password needed), then `supabase migration repair --status applied <version> [<version> …]` (accepts multiple at once). Make the SQL idempotent (`create table if not exists`, `drop policy if exists` before `create policy`, guarded `DO $$ … IF NOT EXISTS` column adds) so a re-run during the session is safe. Verify with a follow-up `db query` count before committing.

## A cross-device realtime sync must subscribe to the table that actually CHANGES, and the subscription must live where it survives navigation (a root-mounted provider), not on the screen that triggered the write.

**Why:** 2026-06-15 a co-driver's stop completions weren't reaching the primary driver's screen in real time. The write path was perfect — `POST /api/complete-stop` updates `dispatch_stops`, RLS lets any authenticated user complete, co-drivers are permitted (Rev 2). The only failure was that the app had **zero** realtime subscription on `dispatch_stops`; the lone route-data channel watched the `routes` table (built for transfer state). A stop completion never touches `routes` mid-route, so nothing told the other client to refetch — it updated only by accident (mount / foreground / the driver's own completion). Two compounding traps: (1) it's easy to assume "we have a realtime channel" covers route data, when that channel is scoped to a different table; (2) the obvious fix — drop the new subscription in the screen where completions happen — would have FAILED silently, because the App Router unmounts page screens on navigation. `DayRouteSelectorScreen` (rendered by `src/app/page.tsx`) unmounts the instant the driver opens a stop, so a screen-scoped channel dies exactly in the bug scenario (driver sitting on StopDetailScreen during the route).

**How to apply:** (a) Identify the table the WRITE lands on and subscribe to THAT table — don't assume an existing channel on a sibling table covers it. (b) Put cross-cutting realtime that must persist across navigation in a root-layout provider (here `AppStateProvider`, which owns `loadDay` + `state.routes` and is mounted once in `layout.tsx`), not in a route screen. Verify mount lifetime: in the Next App Router, anything under `app/<route>/page.tsx` unmounts on navigation; only the root layout and its providers survive. (c) Gate before coding — the table must be in the `supabase_realtime` publication AND have an authenticated-read RLS policy (publication alone is a false green; see the realtime-RLS lesson). (d) Key the effect on the joined ID SET (route ids), not on the row data, so a refetch (which changes stop rows but not the route set) can't re-trigger the subscription. (e) Realtime is false-green-prone — a green build proves nothing; the gate is a live two-device test from every surface, both directions.

## Match the dynamic-route `params` convention of the framework version IN USE, and copy the pattern from a SIBLING page in the same repo — don't assume Next 15 async params on a Next 14 app. `use(params)` on a plain object throws React #438 on every render.

**Why:** 2026-06-15 the dashboard's `/admin/drivers/[driverId]` page crashed 100% of the time with "Application error: a client-side exception" (minified React #438 = "An unsupported type was passed to use(): [object Object]"). The page was authored with the **Next 15** pattern — `params: Promise<{driverId}>` + `const {driverId} = use(params)` — but the dashboard is **Next 14.2.35**, where `params` is a plain synchronous object, not a Promise. `use()` only accepts a Promise or Context, so it rejected the plain object. The bug had been latent since the page was written; it only surfaced when a new feature (the auto-ETA admin toggle) finally routed someone to that page. Every OTHER dynamic page in the repo (`fleet/trucks/[id]`, `fleet/equipment/[id]`, …) correctly used `params: { id: string }` read directly — the page was the lone outlier.

**How to apply:** before writing a dynamic page, check `package.json` for the Next major (`next gen types` / the `params` shape differs: Next ≤14 = plain object, Next 15 = Promise you `await`/`use()`), and grep a sibling `[param]/page.tsx` in the same app for the established pattern — copy it, don't import the pattern from memory or another repo on a newer Next. A build passes either way (the Promise type is structurally accepted at compile time), so this class of bug is **runtime-only** — a green `next build` does NOT catch it. API route handlers (`ctx.params`) and page props can even differ in the same repo; verify the one you're writing.

## To diagnose a "client-side exception" that swallows its own message in production, add a route-level `error.tsx` that RENDERS `error.message` on screen — and put a component-level boundary around the new code so WHERE the error surfaces tells you whether your change caused it.

**Why:** 2026-06-15 a prod page showed only Next's generic "Application error: a client-side exception has occurred (see the browser console)" — and the user's console showed nothing actionable (the crashed page is inert; the throw happened at load, before they opened devtools, and cleared on navigation). Server logs were clean (it was a client render throw), so there was nothing to read server-side. Adding `error.tsx` (a client error boundary that renders `error.message` + `digest` into the page) turned the dead screen into a visible, screenshottable "Minified React error #438" — which decoded (via `raw.githubusercontent.com/facebook/react/main/scripts/error-codes/codes.json`, since `react.dev/errors/438` 404s) straight to the root cause. Critically, I ALSO wrapped the new feature section in a `SectionBoundary`: because the error surfaced through `error.tsx` and NOT the section boundary, that immediately proved the crash was OUTSIDE my change — a pre-existing bug, not the new toggle.

**How to apply:** when a prod page dies to the generic global-error screen and the console is unhelpful, don't keep asking the user to fish for a stack — ship a route `error.tsx` that displays `error.message`/`digest` (in prod, client-component render errors keep their real message; server-component errors are digested). Decode `Minified React error #NNN` from React's `codes.json` on GitHub (the react.dev/errors/NNN page can 404). And isolate blame structurally: wrap your NEW subtree in its own boundary — if the page-level boundary catches it instead of yours, the fault is elsewhere. "Preserve log" + reload-with-devtools-open is the manual fallback, but a rendered error message is faster and the user can just screenshot it.

## A "due today / overdue" queue must key on the DATE column, not on a workflow STATUS that only flips after the fact. If the status transition lags the date, a status-driven list silently undercounts vs. a date-driven one.

**Why:** 2026-06-15 the Will Call driver-app return queue showed "2 orders due back today" while the dashboard board showed many more. The driver app treated "due back" as `status === 'awaiting_return'` only — but an order doesn't flip to `awaiting_return` until the dashboard's overdue process runs (*after* the due date passes), so every order still `picked_up` yet due back today was invisible. Worse, those `picked_up` orders were filtered by `expected_pickup_date` (in the past) → excluded from the Today filter entirely, and grouped into "Out with Customers", never the return queue. The dashboard counted by the date column (`checkin_window_end`), so it was a superset. The correct date helper (`returnByIso` = `checkin_window_end ?? return_reminder_date`) already existed and was wired only to *display*, never to filter/section logic — the data was one function call away the whole time.

**How to apply:** when two surfaces disagree on a "due today / overdue / this week" count, check whether one is keyed on a STATUS enum and the other on a DATE. A status only tells you what a background job has already processed; a date tells you ground truth. For any time-window list, filter and group on the date column and treat the status as orthogonal (it controls the card's *treatment*, not its *membership*). And before adding a date computation, grep for an existing helper — the display layer often already derives exactly the value the filter needs. Same family as "a count and its detail drawer fed by the same collection must broaden together."

## When one status drives MULTIPLE visual cues on a card, splitting that status into a finer sub-state must update EVERY cue — or the card reads contradictory (half-red, half-amber). Watch for cues sourced from a shared atom keyed strictly on the raw status.

## Any loading/auth gate that awaits a network-touching call MUST have a hard, unconditional timeout that resolves the gate — and `setLoading(false)` must sit on a path that runs even if the await never settles.

## Inside a PostgREST `.or(...)`, `like`/`ilike` wildcards are `*`, not `%` — `%foo%` matches literally and returns nothing. For a tiny table, fetch-all + filter-in-JS sidesteps it entirely.

**Why:** Both repos write migrations to the same Supabase project, so the remote `_supabase_migrations` table accumulates rows that neither repo has local migration files for. `supabase db push --linked` refuses to proceed when remote rows aren't present locally — and the CLI's "fix" (`migration repair --status reverted <list>`) would force the OTHER repo to think those migrations need re-running, which fails non-idempotently. That works, but it requires a browser and a paste. 2026-05-14 surfaced a cleaner option: **`supabase db query --linked --file <path>` runs SQL via the Management API and skips the local-vs-remote migration history check entirely.** Same auth (Management API token), same target (the linked project), no history validation. ## CSS `aspect-ratio + width: 100% + max-height: 100%` silently breaks aspect when the parent is shorter than the aspect-derived height.

**Why:** Party Kong's canvas wrapper (2026-05-16 mobile fix arc) used `aspectRatio: W/H + maxWidth: W + maxHeight: 100% + width: 100% + height: auto` to size itself inside a `flex: 1` parent. Intent: "fit inside parent, preserve aspect ratio." Reality on iPhone 14 Pro (parent shorter than the aspect-derived height): browser computes `width: 100%` first (binding constraint, explicit), then `height: auto` via aspect = `width * (H/W) = 720`, then `maxHeight: 100%` clamps to 537. Per CSS spec, aspect-ratio is "preferred" but can be violated by explicit dims — width stays explicit at the clamped 100% value, height stays clamped, aspect-ratio is silently abandoned. Wrapper becomes 390x537 (not aspect-correct). Anything inside that's sized by its own aspect (canvas at 390/720 ratio) then overflows by hundreds of pixels and the wrapper's `overflow: hidden` clips through the visible game area — surfaced as "barely see the top of the guy's head" on commit `891adf4`. Fixed in `b7798bf`.

**How to apply:** When you need a box to fit-and-shrink inside a flex parent while preserving aspect-ratio, drive sizing from the **scarce** dimension. On phones the scarce dim is height, so use `height: 100%, width: auto, aspectRatio: A/B, maxWidth: <px>, maxHeight: <px>`. The browser then uses aspect-ratio to derive width from the height-100% explicit value, and maxWidth/maxHeight cap each end on different viewport sizes (phone → height-bound, desktop → width-bound at maxWidth, derived height capped at maxHeight). The combo of `width: 100% + maxHeight: 100%` is an anti-pattern when both constraints can bind — it works when width is scarce, silently fails when height is scarce.

**Test plan when you set aspect-ratio + max-height on a flex item:** simulate a parent that's both **wider than the aspect-derived width** AND **shorter than the aspect-derived height**. If the box doesn't preserve aspect under that pair, the CSS will fail on some phones. The canonical example: a 390/720-aspect box inside an iPhone-14-Pro-shaped parent (393×537 after toolbars and chrome).

## iOS 18 Writing Tools fires on long-press of any button — held controls drop unless suppressed.

**Why:** iOS 18 introduced a system "Writing Tools" callout triggered by long-pressing any focusable interactive element including `<button>`. ## Canvas internal bitmap (`.width / .height` attributes) and CSS display size are independent — never set inline `canvas.style.width/height` to the bitmap's native dims.

**Why:** Every arcade game's init useEffect did:
```ts
canvas.width  = Math.floor(W * dpr)      // bitmap resolution — correct
canvas.height = Math.floor(H * dpr)      // bitmap resolution — correct
canvas.style.width  = `${W}px`           // ← forces CSS display to native W
canvas.style.height = `${H}px`           // ← forces CSS display to native H
ctx.setTransform(dpr, 0, 0, dpr, 0, 0)   // logical coord space at W×H
```
The last two lines ran post-mount and overrode the JSX's `width: 100%; height: 100%` (since useEffect runs after React's initial render and React doesn't re-set a style prop that hasn't changed). ## The driver-app `tsconfig` targets ES5 — spreading a `Map`/`Set` iterator fails the build. Use `Array.from()`.

**Why:** Fleet Maintenance build (2026-05-22). The first `npx next build` compiled fine but failed type-checking: `Type 'MapIterator<[string, string]>' can only be iterated through when using the '--downlevelIteration' flag or with a '--target' of 'es2015' or higher` on `[...seen.entries()]`. The repo's `tsconfig.json` targets ES5 and does not set `downlevelIteration`, so the spread operator and `for...of` over any non-array iterable (`Set`, `Map`, `Map.keys/values/entries()`) are compile errors. `for...of` over a plain **array** is fine (TS special-cases arrays). `new Map(arrayOfPairs)` and `new Set(array)` are fine (constructor calls, not spreads). Only the `...` spread and `for...of` over the iterator break.

**How to apply:** In this repo, never write `[...someSet]`, `[...someMap]`, or `[...map.entries()]`. Use `Array.from(someSet)` / `Array.from(map.entries())` — `Array.from` is a lib method that accepts any iterable at any target. Same for de-duping: `Array.from(new Set(arr))`, not `[...new Set(arr)]`. If you need `for...of` over a Map/Set, convert with `Array.from` first or use `.forEach`. This is cheap to get right up front and otherwise costs a full build cycle to discover — `next build` compiles before it type-checks, so the error surfaces late.

## On Supabase `onAuthStateChange`, an app-load auth gate must filter to the `INITIAL_SESSION` event — `SIGNED_IN` races a fresh login.

The obvious suspect — `useInspectionStatus`, which fetches `/api/inspection/status` — was a red herring: that hook has **no loading state** (returns only `inspection | null`) and gates no spinner. The real cause was AppState `isLoading`: Home force-reloads on every mount (`loadDay(today, true)`) and gates its whole body on `isLoading`, while `loadDay`'s `fetch('/api/routes')` had **no timeout** and hung offline on iOS standalone (same hang class as the supabase token-refresh). `LOAD_START` set `isLoading=true` and nothing ever resolved it. Routes/StopDetail render from already-present AppState and only call `loadDay` when data is missing (no force, no body-wide `isLoading` gate), so they never exposed the hang.

**How to apply:** When something is "stuck loading," find the *exact* state variable the spinner reads and walk its lifecycle — don't pattern-match to the most recently touched hook. A hook that has no loading state cannot cause a loading hang. Two structural smells make a screen uniquely vulnerable: (a) it force-refetches on every mount, and (b) it gates its whole body on a single `isLoading` flag. Any `fetch` that can run offline MUST be bounded — pair `navigator.onLine === false` (fast-path straight to cache, no doomed request) with an `AbortController` timeout backstop (for the iOS case where `navigator.onLine` lies `true`). `navigator.onLine` is a hint, never the sole gate (see the iOS-onLine lesson above).

## A network-fetch hook that fails closed offline will silently misfire any gate built on its result — scope the gate on `isOfflineMode`, don't trust the fail-closed value.

**Why:** 2026-06-14 stop cards went gray + non-tappable offline. The block was NOT defensive offline code — it was the regulatory pre-trip inspection gate (`stopsLocked`) misfiring: `useInspectionStatus` fails closed to `inspected=false` offline (its fetch rejects), and `!inspected` re-locked an already-inspected route's cards once signal dropped. The instinct "find the offline guard and delete it" would have removed the DOT hard-stop **online** too. The fix was to scope the lock with `!isOfflineMode &&` (offline can't re-confirm; the route cache supports navigation), leaving the online regulatory gate byte-for-byte intact — applied to BOTH `RouteListScreen` and `DayRouteSelectorScreen` (their own comments say "fix both").

**How to apply:** When a UI block appears "only offline," check whether it's a real offline guard or a network-derived value that fails closed (null/false/empty) when the fetch can't run. If the latter, scope the *consumer* on `isOfflineMode` rather than deleting the gate (which usually has an online purpose — regulatory, security, correctness). When the same gate is mirrored across two screens, fix both in the same commit. Belt-and-suspenders option: also make the source hook return a safe sentinel offline so other consumers (CTAs, copy) read correctly too.

## Offline auth restore must cache the user identity yourself — `getSession()` can't restore an expired token offline, and it hangs trying.

**Why:** 2026-06-14 force-close-while-offline → reopen showed the offline `/login` screen. `@supabase/ssr` persists the session in **cookies**; `getSession()` on an **expired** access token calls `_callRefreshToken` (a network call, verified in `GoTrueClient.js`) which hangs/fails offline → no session → `user=null` → every gated page redirects to `/login`. The prior offline restore also *required* a non-expired token, which by reopen had usually lapsed. An expired access token is useless offline anyway (all live queries fail and read from cache), but the **identity** is still known. Fix: mirror the `user` to a fixed-key `ptd_auth_user` cache on every online auth success and restore it synchronously offline, ignoring token expiry, **before** the 3s safety timer (so `loading` never flips false with `user` null — that's the redirect race). Guard a transient offline `null` `INITIAL_SESSION` so it can't clobber the restore; clear the cache on every signOut (4 sites — two bypass the `lib/auth` wrapper).

**How to apply:** For offline-capable auth, don't rely on the SDK's session restore for the expired-token case — cache what you need (user id + roles) yourself, restore it synchronously on an offline cold-start, and gate it on your own invariants (here: the day-change `ptr_session_date` check). Audit EVERY signOut path when adding an identity cache — direct `supabase.auth.signOut()` calls bypass a wrapper-level clear; centralize in the wrapper AND clear explicitly at the direct sites.
